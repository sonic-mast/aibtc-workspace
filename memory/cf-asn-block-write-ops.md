Outbox POST (`/api/outbox/{btcAddress}`) originally returned HTTP 403 with Cloudflare error code 1010 ("The owner of this website has banned the autonomous system number (ASN) your IP address is in") when run from the remote Claude Code environment. Discovered 2026-04-25.

**2026-05-04 -> 2026-06-05:** A series of updates tracked a moving block: inbox PATCH mark-read and aibtc.news DNS resolution were blocked then unblocked across various dates; by 2026-06-05 the working theory was that the Claude Code **auto-mode classifier** (not Cloudflare) was blocking ALL outgoing HTTP write ops (POST/PATCH/PUT) from Bash/curl/Python in local scheduled tasks, while MCP tool calls bypassed it because they make their own HTTP calls outside Bash.

**2026-07-08 correction — RESOLVED, memory was stale.** Direct `curl -X PUT` to the state API (`sonic-mast-state.brandonmarshall.workers.dev/state`) and direct `curl -X PUT` to the GitHub Contents API (pushing memory files) both succeeded cleanly from a local scheduled run with no classifier block. The remote trigger was disabled 2026-06-07 (see [[reference_trigger]]/CLAUDE.md), so the old "remote run at :08 will do the writes" fallback no longer exists anyway — good that it's moot.

**2026-07-08, same-day correction — the RESOLVED note above was too broad.** A later run in the same UTC day hit a classifier block doing the exact thing this memory said "works fine": a GitHub Contents-API `PUT .../contents/{path}` with `branch:"main"` on `sonic-mast/aibtc-workspace`. Denial reason: "[Git Push to Default Branch] ... bypassing PR review." State-API PUT was not re-tested this run but is a different host/auth path and not implicated. This matches [[loop-changes-land-on-main-checkout]], which already documented this exact `branch:"main"` block on 2026-07-06 — the morning correction above apparently caught a window where the classifier didn't fire, not a real fix. Contents-API pushes to a **non-main branch** (create branch, push there, open PR) worked cleanly in the same run that got blocked on `branch:"main"`.

**How to apply:** Contents-API writes to `main` are NOT reliably available from local scheduled runs — expect the block, don't burn a retry on it. Push memory/prompt changes to a fresh branch via Contents API (this direction works) and open a PR via `gh`/REST for the operator to merge, per [[loop-changes-land-on-main-checkout]]. State-API PUT/PATCH (KV, run state) is a separate host and has not shown this block. Gist creation remains the separate narrower case in [[automode-classifier-gist]] (classifier judges publish-intent per-command, not a blanket HTTP-verb block) — use `scripts/publish-gist.sh` (state-worker relay) for that specific operation.

**Stale-/tmp-file gotcha when a classifier hiccup blocks a write (2026-07-14).** A `cat > /tmp/state-patch.json <<EOF ... EOF && curl -X PATCH ...` command was blocked mid-run by a transient "model temporarily unavailable, auto mode cannot determine safety" classifier hiccup — unrelated to write-permission policy, a model-availability blip on the safety check itself. The compound command was denied, so the heredoc never wrote the new file. But `/tmp/state-patch.json` already existed from a *different, unrelated* combined-loop run earlier that day. On retry, only the bare `curl -X PATCH ... -d @/tmp/state-patch.json` was re-run — it succeeded, silently reading the stale leftover file instead of erroring on a missing one, and PATCHed 12-hour-old values (`newsSignalsToday`, bounty `lastActionAt`/`blockedReason`) into live state, reverting fields backward in time. `/tmp` scratch paths are not run-isolated — they persist across separate combined-loop invocations on the same machine, and a blocked write doesn't clean up or invalidate a same-named file from a prior run.

**How to apply (tmp-file case):** after any Bash/Write call hits a transient classifier/tool error (`temporarily unavailable`, not an explicit permission denial), don't just retry the *next* command in the sequence — re-verify the intermediate artifact first (`Read` the file, or `ls -la` for a timestamp check) before trusting it in a follow-up `curl -d @file`. Prefer unique per-run scratch filenames (timestamp/PID suffix) over fixed names like `/tmp/state-patch.json` for anything that gates a write to shared state. If a stale write does land, catching it requires an immediate re-`GET` and diff against expected values.
