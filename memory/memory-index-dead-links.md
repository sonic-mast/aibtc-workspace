description: Repo memory/MEMORY.md had ~20 links to files that never existed in this repo's git history; fixed 2026-07-05
type: reference
---

`memory/MEMORY.md` (the index the combined loop reads each run) referenced ~20 files — `user_brandon.md`, `feedback_commit_push.md`, `feedback_no_fabrication.md`, `feedback_disclosure_separate.md`, `feedback_sync_fork.md`, `feedback_news_api_status_decoupled.md`, `feedback_quantum_governance_signals.md`, `feedback_quantum_hardware_vs_projection.md`, `feedback_twitter_source_insufficient.md`, `feedback_quantum_google_derivative.md`, `feedback_cross_agent_dedup.md`, `feedback_bitcoin_macro_edgar_anchor.md`, `feedback_discussions_participation.md`, `feedback_mcp_no_npx.md`, `feedback_correction_dedup.md`, `project_dual_stacking_enrolled.md`, `project_bff_skills.md`, `reference_trigger.md`, `news_scoring_dimensions.md`, `reference_shelly_is_sonic_mast.md` — none of which exist under `memory/` and `git log --all -- memory/<name>.md` returns empty for each, meaning they were never committed to this repo at all.

**Why:** Cause unconfirmed. The dead-link filenames match almost 1:1 with entries in a *different* memory system — the Claude Code harness's own cross-session auto-memory store (outside this repo, under `~/.claude/projects/.../memory/`), which is unrelated to this repo's `memory/` folder but happens to track the same agent/project. Best guess: at some point index content was copied from that system into this repo's `MEMORY.md` without ever creating the matching files here, or a past `push_mem` call updated `MEMORY.md` but silently failed to push the paired memory file.

**Fixed 2026-07-05:** 3 entries had real content under renamed files and were re-linked (`feedback_outbox_field.md`→`outbox-field.md`, `feedback_inbox_field_names.md`→`inbox-field-names.md`, `feedback_wallet_unlock_literal.md`→`wallet-unlock-env-expansion.md`). `project_eic_pause.md`'s index one-liner was also stale relative to its own file content (said "still paused as of 2026-06-15"; the file itself already said "resumed 2026-06-24") — corrected. The remaining ~15 dead entries were dropped from the index rather than reconstructed, since their content can't be verified and several are already covered elsewhere (fork-sync, disclosure-separation, and no-fabrication rules are all in CLAUDE.md; quantum/twitter/cross-agent-dedup/EDGAR gates are all in automation-prompts/aibtc-combined.md's gate tables; Discussions sweep and BFF-contest-ended status are in aibtc-combined.md Phase 2b.1 / Phase 5a). A few (project_dual_stacking_enrolled's deployment-required nuance, reference_shelly_is_sonic_mast, feedback_news_api_status_decoupled) may represent genuinely lost operational knowledge with no surviving source — flagged to the operator via run-log `notable` rather than unilaterally reconstructed.

**How to apply:** Before trusting any `MEMORY.md` line, or before writing a new one, check the target file actually exists under `memory/` (`ls memory/` or the repo tree) rather than assuming the index is accurate — it has drifted from reality before without any single run noticing. When several unlinked `.md` files exist under `memory/` with no index entry (there were ~28 at last check — e.g. `arxiv-permission-denied.md`, `bitflow-ticker-empty.md`, `identity-service-extended-outage.md`), that's a lower-severity version of the same drift (real content, just not discoverable) — lower priority than dead links but still worth reconciling opportunistically.
